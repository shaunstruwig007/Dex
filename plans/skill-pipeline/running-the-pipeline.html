Running the pipeline — idea to build handoff, with confirm beats.

Three paste-ready starter prompts for the most common situations — cold start, continuing mid-flow, and dry-running for shape validation. Plus a per-stage checklist of what to type, what to expect back, and what to scan before moving to the next stage.

Surface: Claude Code interface
Pairs with: pipeline-flow.html
Skill catalogue: .claude/skills/
Operator: the human at the keyboard

Three starter prompts — pick the one that matches your situation

Paste-ready. Replace the placeholders, then send. The pipeline runs from your Claude Code chat — skills are slash-commands, artefacts land in the vault, you stay in the driver’s seat.

Use when

Cold start

A fresh idea. You want Dex to drive the full pipeline with confirm-before-proceed beats between stages.

I have a new product idea to take through the
skill pipeline.

**Idea:** <one paragraph — the spark, not the spec>

Run the pipeline with confirm-before-proceed beats
between stages:

1. /initiative-discovery-custom on this idea.
   Show me the discovery package + evidence gaps
   + Slice-1 candidate confidence. Pause.

2. /prd-author-custom on the discovery.
   Show me the slices table + plan-mode-seed
   + open questions. Pause.

3. Critique pair — /critique-product-custom
   first, then /critique-engineering-custom.
   Show verdict per row. If RETURN TO PLAN, list
   must-fixes and stop.

4. I’ll tell you which must-fixes to fold.
   /prd-author-custom --rerun, then re-run
   the critique pair. Loop until READY FOR BUILD
   (max 3 passes).

5. On READY FOR BUILD, give me the Build handoff
   block as a copy-paste I can take to the codebase
   repo’s Cursor Plan mode.

Cite vault paths when surfacing evidence. Don’t
invent metrics or stakeholders not in the vault.
If a skill refuses, tell me which refusal triggered
and why — don’t auto-recover.

Beat structure: 5 hand-offs from you, ~5–10 minutes wall-clock per stage. Two critique passes is normal — three is a smell.

Use when

Continuation

You’re mid-flow. Discovery (or PRD, or critique) is done. You want to take it one stage further and stop for review.

Discovery’s done — file is at
06-Resources/Product_ideas/<slug>_discovery.md.

Run /prd-author-custom against it.
Stop after the draft so I can scan the slices
table + plan-mode-seed before we critique.

Most realistic shape. Replace the file path + the skill + the “stop after” instruction for whichever stage you’re progressing to. Keeps you in the driver’s seat without re-loading prior context.

Use when

Dry-run

Validating the pipeline shape after a skill change, after a vault refresh, or for a teammate’s first walkthrough. You want to see the shape, not ship the artefact.

Dry-run the skill pipeline on this canned idea:
<something deliberately simple, e.g. “let employees
pin a post to their own profile”>

Don’t write files yet — show me what each skill
would produce + where it would refuse + which open
questions it’d surface. I’m validating shape, not
shipping.

Smoke test. Useful after the catalogue changes (a skill rename, a new refusal, a tightened input contract). Cheap to run; surfaces breakage early.

What to expect at each stage — and what to scan before continuing

The same shape across all 7 stages: You type (or You do for stages without a skill) · What Dex returns · Before you proceed, scan for.

Stage 01

Idea

no skill — you write

The spark. Captured as a paragraph, not a spec. Grain check: one user, one problem, one observable shift if it shipped.

You do

Write the idea down — one paragraph in chat, in a backlog row, or on a sticky. No structure required.

What Dex returns

Nothing — this stage is for you. The pipeline starts when you invoke the next skill.

Before you proceed, scan for
  • One idea, not three. If it’s three, split before discovery starts.
  • Names a user, a problem, or an outcome. One of those is enough.
  • Could a skeptical engineer-friend ask “what’s the smallest version?” and you have an answer? If yes, proceed.
Stage 02

Discovery

/initiative-discovery-custom ↗

Vault scan with relevance confirmation, ICP cross-check, customer-evidence harvest, candidate slicing. Produces a discovery package — the input the PRD author refuses to draft without.

You type /initiative-discovery-custom "<one-line idea>"

Or paste the idea paragraph and ask Dex to run discovery against it.

What Dex returns

Discovery file at 06-Resources/Product_ideas/<slug>_discovery.md + chat summary covering: candidate Slice 1 + walking-skeleton confidence, evidence-gap count, open-question count, exec-roadmap match, Friday Signal freshness.

Before you proceed, scan for
  • Exec-roadmap match? Matched = exco-validated, scope & de-risk only. Off-roadmap = needs surfacing first.
  • Evidence gaps explicit? Sparse-vault is honest. Fabricated conviction is bad.
  • Candidate Slice 1 reads as a one-sentence demo you could stand behind in front of stakeholders?
  • Stakeholders named? Anonymous “the VP of Sales” is a gap.
  • Friday Signal current (or stale-warning acknowledged)?
Stage 03

PRD

/prd-author-custom ↗

Drafts a slice-shaped bond_v1 PRD. Idempotent — your edits are signal, never silently overwritten on re-runs.

You type /prd-author-custom <discovery-file>

Or just /prd-author-custom if the discovery file is the active context. Use --rerun after folding critique must-fixes.

What Dex returns

PRD at 06-Resources/PRDs/<Feature>.md + chat summary covering: number of slices, Slice 1 walking-skeleton confidence, open-question count, suggestion to run the critique pair next.

Before you proceed, scan for
  • Slice 1 = walking skeleton? Touches all layers needed for an end-to-end demo. One observable outcome.
  • Plan-mode-seed block reads as something you’d paste into Cursor Plan mode without edits?
  • Test shape per slice present (5b)? Required since walkthrough-2.
  • Slice 1 demo readiness checklist (5c) protects the first steerco moment?
  • Every risk has a mitigation? A risk without one is incomplete.
  • Body Status line in sync with frontmatter lifecycle: + critique_status:?
Stage 04

Critique

/critique-product-custom ↗ · /critique-engineering-custom ↗

Pair-protocol: product runs first (frames the why), engineering runs second (sharpens the how). One unresolved GAP = RETURN TO PLAN.

You type /critique-product-custom /critique-engineering-custom

Run product first. Wait for the verdict. Then run engineering. Or ask Dex to run the pair in sequence, with verdicts shown after each.

What Dex returns (per skill)

Scorecard with PASS / SOFT / GAP per row — 7 rows for product, 9 rows for engineering — plus must-fixes (P1, P2… or E1, E2…), nice-to-haves, and verdict (READY FOR BUILD or RETURN TO PLAN). Engineering also returns eng-alt alternatives (always ≥1).

Before you proceed, scan for
  • Verdict — READY FOR BUILD or RETURN TO PLAN?
  • Any GAP that surprises you? A GAP is a real blocker. SOFTs are recommendations.
  • Any eng-alt worth considering? Cheaper-path is the highest-yield row across critiques.
  • Decide: fold the must-fixes (back to stage 03 with --rerun), override one (note the trade-off in the PRD body), or stop?
The critique loop. Must-fixes → /prd-author-custom --rerun (idempotent — preserves your edits) → re-run the critique pair. Two passes is normal. Three is a smell — something deeper is wrong with the brief or the slicing.
Stage 05

spec_ready

no skill — you confirm

Not a skill. A state. The PRD has earned the “safe to design against, safe to slice into tickets” label. Verifying frontmatter + body Status line agree is your beat here.

You do

Open the PRD file. Confirm the frontmatter and the body Status line match. If they don’t, ask the PRD author skill to update the body Status line.

What Dex returns

Nothing — this is a state-confirm beat, not a skill invocation. (If you ask Dex to verify, it’ll read the file and tell you if anything’s out of sync.)

Before you proceed, scan for
  • Frontmatter: lifecycle: spec_ready + critique_status: must_fixes_folded?
  • Body Status line matches the frontmatter? (External reviewers shouldn’t need to parse YAML.)
  • Build handoff section (section 11) is filled, not [TBD]?
  • Sibling PRDs in related_prds: are themselves spec_ready if Slice 1 has cross-PRD dependencies?
Stage 06

Design

vendor tool — Claude Design / Figma

Designer takes the PRD’s Design pointers + A11y subsection + Failure-mode UX as the brief. No Dex skill in this loop — design tools live elsewhere. (Design’s typed role across the pipeline is broader than this stage; see design-in-the-pipeline.html.)

You do

Hand the PRD to the designer (or run Claude Design wireframe mode yourself if you’re solo). Point them at Design pointers, Failure-mode UX, and the A11y subsection. Update design_pass_status: in the PRD frontmatter.

What Dex returns

Nothing in this catalogue — design happens in Claude Design / Figma / hand-drawn. Dex re-engages when wireframes are reviewed and any ambiguities loop back as PRD edits.

Before you proceed, scan for
  • Design pointers read as a clean handover (no PM-overreach into design decisions)?
  • A11y has the lang-attribute requirement if there’s translated content?
  • Design slice ordering matches the build slice ordering?
  • What designer should NOT prescribe (decisions belonging to PM / engineering) is explicit?
Stage 07

Build handoff

Cursor Plan mode · codebase repo

PRD jumps repos: from the GitHub vault to the Bitbucket codebase. Engineer pastes the plan-mode-seed block into Cursor Plan mode — one line per slice becomes one Plan-mode step becomes one PR.

You do

Copy the PRD to the codebase repo at docs/PRDs/<filename>.md. Open Cursor with the codebase repo as workspace, with the PRD(s) in context. Paste the plan-mode-seed fence as the Plan mode prompt.

What Dex returns

Nothing in this catalogue — Cursor Plan mode takes over from here. (If the PRD needs spec changes, edit the source in the vault and re-run /prd-author-custom; the codebase copy is a snapshot, not a live source.)

Before you proceed, scan for
  • Sibling PRDs that need to come along (cross-PRD slice dependencies)?
  • Hard gates before Slice 1 build (blocking open Qs)?
  • Sign-offs needed before Build (Product / Engineering / Legal / Design / domain-owner)?
  • Source-of-truth rule understood — codebase copy edits don’t propagate back to the vault?
