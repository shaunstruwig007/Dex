Skill in the pipeline · stage 04 (product half)

Critique — Product — is this product-honest before we build?

Reads a draft plan, PRD, or sprint seed and returns a 7-row scorecard — PASS / SOFT / GAP — with concrete must-fixes the author can fold without redesigning. Catches outcome-integrity gaps and first-demo risk before they become a steerco moment. Pairs with /critique-engineering-custom.

The 7-row checklist — every artefact runs against this

# Row The skill asks GAP if…
01 Outcome integrity What user/business outcome shifts when this ships? Will the first real Steerco / dogfood / customer-visible moment notice? Artefact ships craft without changing an observable user moment. Slice 1’s “demo” is to engineering, not a stakeholder, but the artefact frames it as customer-noticeable.
02 First-demo risk If we demoed this on a real initiative tomorrow, what’s the most embarrassing thing that could happen in the room? Is it covered by demo prep? Risks section is technical (covered by engineering critique) but the actual-demo content is unprepared. E.g. “translation quality could embarrass us in front of execs” with no script for which posts to demo on.
03 Honesty of asks Does the artefact ask the user (PM, Steerco, designer) for evidence they actually have at this stage of the journey? Artefact asks for data the user doesn’t yet have. E.g. “engagement parity within 15% within 60 days” with no baseline measurement in place.
04 Cohesion vs craft Does the artefact move the journey forward as a whole, or just polish one surface while the next is broken? One screen gets love; the seam to the next column / surface is ignored. Translated post + untranslated comments below it = visible seam.
05 Bar discipline MVP bar vs next-bar vs Phase 2 — anything bleeding upward in scope? Any vague “audit then wire” that could explode? Next-bar / Phase-2 work has crept into the current artefact without an ADR or scope-lock.
06 Out-of-scope hygiene Are out-of-scope items protected? Will any of them get pulled in mid-Build? Does the artefact silently modify a sibling PRD’s surface? Artefact lists items as out-of-scope but the body adds them anyway. Or modifies another PRD’s feature surface without a coordination note to that PRD’s owner.
07 Reversibility If this turns out wrong in a week, can we back out cleanly? Or do we own a one-way door? Artefact introduces irreversible UX habits / data shapes / vendor lock-in without an ADR or mitigation.

The load-bearing rule — one GAP blocks the build

The skill never ships “mostly ok” — one unresolved GAP = RETURN TO PLAN

Every row scores PASS, SOFT, or GAP. SOFTs get a must-fix recommendation but don’t block. GAPs block — the artefact returns to the PRD author, who folds the must-fix and re-runs the critique. The loop closes when every row is PASS or SOFT.

A “mostly ok” verdict trains the team to ignore the scorecard. The hard one-GAP rule keeps the critique meaningful — if the skill says READY FOR BUILD, that means something. Two passes is normal. Three is a smell — if the loop runs three times, the underlying brief or PRD is the problem, not the slice shape.

Pair protocol — product runs first, engineering runs second

Why this order?

Product reframes intent (“what are we shipping, and does it move a user moment?”). Engineering then tells us if the reframe is buildable. If engineering goes first, alternatives sometimes optimise for a thing product would have killed.

1. Run first

/critique-product-custom

Frames the why — outcome, demo risk, honesty, cohesion, scope, reversibility.

→

2. Run second

/critique-engineering-custom

Sharpens the how — atomicity, idempotence, schema, contracts, failure modes, a11y, tests, blast radius, cheaper paths.

→

3. Author folds

Net product + eng applied

Both must-fix sets in a single edit, under one heading. PRD re-runs critique on demand.

Conflict resolution: if the two critiques disagree, the artefact author picks one and notes the trade-off in the artefact body. Neither skill has a veto.

Anti-patterns the skill flags on sight

Anti-patterns · never silent

  • “We’ll add the test later” on a flow that’s about to be the demo.
  • A required field that the user has to lie to answer.
  • A summary screen that just lists fields back without synthesis.
  • Two buttons whose copy doesn’t tell the user what is different.
  • A drag affordance with no keyboard story.
  • Density / view toggles with no persistence.
  • Plans that say “out of scope” but quietly add the scope to a deliverable.
  • Re-asking a question the previous skill / step already answered.
  • “We’ll backfill the docs” — same-PR documentation discipline violation in waiting.
  • A success metric whose baseline doesn’t exist in the system today.
  • Slice 1 framed as customer-noticeable when it’s actually engineering-only.

Hard refusals — what the skill will not do

Refusals

Never bypassed

  • Critique without reading the artefact end-to-end. Read first, then talk.
  • Replace engineering critique. That’s /critique-engineering-custom’s lane.
  • Propose code. The skill decides whether the artefact is product-honest, not how to fix the code.
  • Bless an artefact with an open GAP. Return it to plan.

Voice discipline

How it talks

  • Lead with the verdict. One sentence per row. Quote the file + line when blocking.
  • No platitudes. Never “looks good overall” or “this is a strong start.”
  • Direct, opinionated, never operational. Decides the call — doesn’t tell engineering how to write the code.
  • Voice example: “Honesty of asks: GAP — the brief asks for engagement-parity baseline; the team has no baseline measurement in place. Drop the metric or replace with one measurable at slice ship.”
