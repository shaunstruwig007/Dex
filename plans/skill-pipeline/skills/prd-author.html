Skill in the pipeline · stage 03

PRD Author — discovery becomes a slice-shaped spec.

Reads a discovery output and writes a bond_v1 PRD — goal, metrics, requirements, walking-skeleton + thickening slices, test shape per slice, demo-readiness checklist, and a plan-mode-seed block that maps 1:1 to Cursor Plan mode. Idempotent — the author’s edits are signal, never silently overwritten.

The bond_v1 shape — frontmatter + 11 sections, fixed order

prd_shape: bond_v1 prd_id: <slug>-YYYY-MM-DD created_date: YYYY-MM-DD last_bond_run: YYYY-MM-DD HH:MM lifecycle: spec_ready # idea | discovery | design | spec_ready | develop | uat | deployed | parked critique_status: pending # pending | running | must_fixes_pending | must_fixes_folded design_pass_status: not_applicable # not_applicable | pending | scheduled | in_progress | done related_prds: [<Sibling_PRD>.md, ...]
01Goal

One paragraph: what shifts when this ships. One paragraph: the evidence grounding (sales, customer, ICP, signal). No implementation detail.

02Users

Primary user (narrative). Secondary users (narrative each). Out-of-scope user roles with reasoning.

03Success metrics

2–4 metrics. Each: name, definition, target with timeframe, measurement source. Plus a Note on measurability — catches aspirational-metric anti-pattern.

04Requirements

Outcome-shaped, numbered. No implementation prescription. One sentence each, user-observable behaviour. 5–10 typical.

05Slices

The heart of the PRD. Table: # · Name · Walking-skeleton? · Demo outcome · Layers · Depends on. Slice 1 = walking skeleton, end-to-end, observable in one sentence.

5bTest shape per slice

Required subsection. Table: unit / integration / e2e / a11y per slice. PR review enforces. Promoted from per-PRD must-fix → required by skill 2026-04-29.

5cSlice 1 demo readiness

Required checklist. Pre-vetted demo content · dependency-failure fallback rehearsed · defect example off-camera · wireframe walkthrough if Slice 1 is engineering-gated. Protects the first steerco demo.

06Plan mode seed

Fenced plan-mode-seed block, mechanically generated from the Slices table. One line per slice. 1:1 paste into Cursor Plan mode — the load-bearing contract.

07Risks

4–8 risks. Each: name, description, mitigation (required, not optional). PRDs with external deps add a Technical failure modes subsection.

7′Technical failure modes

Required subsection when the PRD touches vendors, APIs, realtime stacks, or external datasources. Idempotency, cache, timeout, vendor lock-in. Promoted to required 2026-04-29.

08Out of scope

Table: out-of-scope · why · future cycle? Each row = explicit exclusion + reason + return-window flag.

09Open questions

Table: question · blocks · owner / next step. Followed by a “None of these block slice-1 build” subsection with stub assumptions.

10Design pointers

Paste-ready for Claude Design / a human designer — no skill in the loop. Subsections: context, surfaces, critical UX questions, failure-mode UX, a11y, constraints, what NOT to prescribe, design slice ordering.

10′A11y subsection

Non-negotiable. lang attribute on translated content · keyboard parity · aria-live · contrast for outdoor / low-light · tap targets. Promoted to required 2026-04-29.

11Build handoff

Required when vault-PRD & codebase live in different repos. Repo-split callout · how-to-use in Plan mode · handoff snapshot table · source-of-truth rule. Added 2026-04-29.

The load-bearing rule — idempotence on the PRD file

Author edits are signal — the skill never silently overwrites

Before writing, the skill checks if the target file already exists and compares the file mtime against the frontmatter last_bond_run:

  • File doesn’t exist → write fresh.
  • File exists, no human edits since last run (mtime matches last_bond_run) → safe to overwrite. Write directly.
  • File exists, edited since last run → stop. Surface a diff. Ask: Overwrite (lose edits) · Merge (skill picks edits where they conflict; user reviews) · Save-as (write to <filename>.bond.md) · Abort.
Without this rule, every --rerun would clobber the author’s manual revisions. The PRD would degrade with every refresh. Idempotence makes the skill safe to invoke without fear — even on a heavily-edited PRD.

Required input contract — the skill refuses to draft without one of these

1. Discovery output (default path)

Output of /initiative-discovery-custom. Required fields:

  • discovery.problemStatement, discovery.userSegments[] (with is_primary), discovery.stakeholders[]
  • discovery.evidenceLog[], discovery.evidenceGaps[], discovery.icpSegmentMatch
  • discovery.candidateSlices[] · discovery.openQuestions[] (with blocks + owner)
2. --no-discovery override

Requires a one-line reason (e.g. “migrating legacy PRD; original discovery is the existing PRD body”). Reason gets logged in frontmatter as discovery_skipped_reason:.

3. --reshape <existing-prd>

Input is the existing PRD file. Skill extracts what it can (Goal, Users, Risks) and asks the user to fill any missing sections inline. Used to migrate legacy PRDs to the bond_v1 shape.

If none of the three apply, the skill stops and prompts the user to run /initiative-discovery-custom first.

How to invoke

/prd-author-custom                              # author from active discovery output
/prd-author-custom <discovery-file>             # author from a specific discovery file
/prd-author-custom --reshape <existing-prd>     # migrate an existing PRD to bond_v1 shape
/prd-author-custom --no-discovery               # author without discovery (REQUIRES reason logged in frontmatter)

Hard refusals — the skill stops and asks before generating

Refusals · structural

Stops cold on these

  • No discovery output and no --no-discovery reason → “Run /initiative-discovery-custom first.”
  • Existing PRD edited since last run → diff + ask before overwriting.
  • A metric with no measurement source → “Name the source or drop the metric.”
  • A risk with no mitigation → “Add one or remove the risk.”
  • A slice with no observable demo outcome → “If it’s a refactor, fold into the slice it supports.”

Refusals · cross-cutting

Walkthrough-2 must-fixes, enforced

  • Slice 1 not a walking skeleton → “Touches only <layers>. Thicken slice 1 or move the missing layer in.”
  • Test shape per slice missing → “Cross-cutting must-fix. Fill or mark n/a with reason.”
  • Slice 1 demo readiness checklist missing → “First-demo risk is unprotected.”
  • External-dependency PRD with no Technical failure modes subsection → “Spec at minimum: unavailable, idempotency, vendor lock-in.”
  • Translated content with no lang attribute requirement → “Non-negotiable for screen readers.”
  • Build handoff section missing when vault & codebase repos differ → “Add the repo-split contract.”
  • Body Status line out of sync with frontmatter → “Update body so external reviewers see pipeline state without parsing YAML.”

Cross-cutting findings — promoted into bond_v1 on 2026-04-29

Walkthrough-2 evidence · folded once, applies forever

Six findings that became required-by-skill

  • Test shape per slice — was a GAP on every PRD critiqued in walkthrough-2. Now required (section 5b).
  • Slice 1 demo readiness — was SOFT on every product critique. Now required (section 5c).
  • Technical failure modes subsection in Risks — was GAP on engineering-heavy PRDs. Required for any PRD with external dependencies.
  • A11y subsection in Design pointers — the lang attribute on translated content was the original walkthrough-1 catch. Non-negotiable.
  • Build handoff section — addresses the GitHub-vault → codebase-repo split. Without it, the developer has no operational pickup contract.
  • Body Status line in sync with frontmatter — surfaces pipeline state to readers without expanding YAML.

Cross-cutting findings ledger: plans/skill-pipeline/sessions/2026-04-29-walkthrough-2-critiques.md.

Coexists with /agent-prd — different shape, different downstream consumer

Two PRD authors live in the catalogue. They serve different loops — pick by downstream consumer.

  • /prd-author-custom — this skill. Slice + plan-mode-seed shape, for Cursor Plan mode in a discovery→PRD→build pipeline. Idempotent. Refuses to draft without discovery.
  • /agent-prd — wizard-driven, work-package + BDD shape, for direct agent implementation. Different downstream consumer.
