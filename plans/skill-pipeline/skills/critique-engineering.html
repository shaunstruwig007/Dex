Skill in the pipeline · stage 04 (engineering half)

Critique — Engineering — is this buildable, safe, and the right shape?

Reads a draft plan, PRD, or sprint seed and returns a 9-row scorecard — PASS / SOFT / GAP — with concrete must-fixes and at least one cheaper alternative shape. Catches contract drift, atomicity gaps, and a11y misses before they hit runtime. Pairs with /critique-product-custom.

Skill: /critique-engineering-custom
Replaces: /agent-q-cto-custom

Where it sits in the pipeline

Pipeline stages 01 Idea → 02 Discovery → 03 PRD → 04 Critique → 05 spec_ready → 06 Design → 07 Build

Two lenses — same checklist, different rigour

Runtime-plan lens

Path: plans/PDLC_UI/sprint-N-seed.md

Applies when the artefact is a runtime / sprint plan. Schema parity means literal Zod / migration / fixture parity. Atomicity means literal database transactions. Contracts are shipped API event payloads.

  • Reads code, not just docs — pulls in affected runtime files in pdlc-ui/src/.
  • Honours ratified ADRs in pdlc-ui/docs/adr/ — never silently violates or supersedes.

Anchors: plan.md · engineering-guardrails.md · tech-stack.md · schema-initiative-v0.md

Feature-PRD lens

Path: 06-Resources/PRDs/<feature>.md

Applies when the artefact is a feature PRD. Schema parity means data contract coherence across consumers. Atomicity means multi-write user-action paths. Contracts are the API surface the PRD implies.

  • Verifies fields the PRD assumes “exist conceptually” actually exist in production schema / migrations.
  • Reads related_prds: frontmatter to understand surface intersections.

Anchors: linked PRDs · existing schema files / DB migrations

The skill picks the lens automatically: runtime files cited or path under plans/PDLC_UI/ → runtime lens · PRD frontmatter or path under 06-Resources/PRDs/ → feature-PRD lens.

The 9-row checklist — every artefact runs against this

# Row The skill asks GAP if…
01 Atomicity Does any user action need two writes that can drift? Save→transition chain instead of a single transaction. Multi-write publish path with no specified behaviour on partial failure.
02 Idempotence What happens if the request runs twice (network retry, double-click, replay)? No server-side guard; identical replays cause duplicate state. Cache key shape that allows duplicates.
03 Schema parity Runtime lens: Do schema doc, runtime Zod, golden fixture, and migrations all agree after this PR? PRD lens: Does the data contract make sense across all consumers — and do fields the artefact assumes exist actually exist in production? One of the four lags (runtime). PRD assumes a column / field exists “conceptually” but it’s not in the schema (PRD).
04 Contracts unchanged Does the artefact touch a previously-shipped contract (event payload, API shape, gate function, post payload)? Contract widens or narrows silently — no ADR, no doc update, no consumer enumeration.
05 Failure modes What does the user see when the network / server / DB / 3rd-party API fails? Backoff? Retry? Visible degraded state? Artefact only covers the happy path. No prescribed behaviour for the obvious failure cases.
06 A11y first-class Keyboard path, focus ring, screen-reader semantics, axe story? lang attribute on translated content? Touch-target sizes? A11y is a “follow-up” or only on the menu/fallback path. Translated content rendered without the lang attribute (screen reader mispronounces). New interactive element with no keyboard parity.
07 Test shape Unit + integration + e2e + a11y — proportional to the change? Per slice, what’s the test coverage? One layer carries the weight (often e2e); unit gaps will cost on CI flakes. PRD has no test shape per slice at all.
08 Reversibility / blast radius If wrong, what’s the back-out? Data migration needed? Vendor lock-in? Artefact introduces an irreversible shape (vendor, schema, UX habit) without an ADR or a migration / mitigation story.
09 Cheaper path Is there a smaller, sturdier way to hit the same outcome? This row never produces a GAP. Always produces ≥1 alternative shape labelled eng-alt. Up to three. Mandatory — even if the answer is “current shape is the simplest path,” say it explicitly.

The load-bearing rule — eng-alt is mandatory

Row 9 (cheaper path) always produces at least one alternative shape — even if the answer is “current shape is simplest”

The author is free to ignore the alternatives, but they have to be offered. Up to three eng-alts per critique. The skill cannot skip this row — it’s the consistently-highest-yield row across critiques.

Without a forced alternative, the engineering critique becomes a checklist. With it, every PRD gets at least one moment of “is there a smaller way?” explicitly considered. Even “no, current shape is the simplest” is a useful answer — it’s an explicit decision, not a default.

Pair protocol — engineering runs second

Why product runs first?

Product reframes intent. Engineering then tells us if the reframe is buildable. If engineering goes first, alternatives sometimes optimise for a thing product would have killed. Engineering critique is sharper when the why is locked.

1. Run first

/critique-product-custom

Frames the why — outcome, demo risk, honesty, cohesion, scope, reversibility.

→

2. Run second

/critique-engineering-custom

Sharpens the how — atomicity, idempotence, schema, contracts, failure, a11y, tests, blast radius, cheaper paths.

→

3. Author folds

Net product + eng applied

Both must-fix sets in a single edit, under one heading. PRD re-runs critique on demand.

Conflict resolution: if the two critiques disagree, the artefact author picks one and notes the trade-off in the artefact body. Neither skill has a veto.

Anti-patterns the skill flags on sight

Anti-patterns · never silent

  • Two-API chains that should be one transaction.
  • Client-only validation guarding a state change (no server enforcement).
  • A new event kind without same-PR Zod + doc + fixture update.
  • setInterval / setTimeout / after() in route handlers when the runtime is tick-driven.
  • New dependency for a problem solvable with existing primitives.
  • HTML5 drag-and-drop for any flow that needs keyboard parity.
  • Translated content rendered without the lang attribute on the wrapping element.
  • New table without (<entity_id>, <kind>, started_at DESC)-style index when the obvious query needs it.
  • TODO(Sn) markers added in this sprint with no DoD line owning them.
  • Test only covers e2e — no unit / integration; will flake on CI.
  • “We’ll handle the edge case in S<n+1>” with no DoD entry in S<n+1>.
  • PRD assumes a database column or API field exists without checking.
  • API payload change with no consumer enumeration.

Hard refusals — what the skill will not do

Refusals

Never bypassed

  • Critique without reading end-to-end — including the affected runtime files when the runtime lens applies.
  • Replace product critique. That’s /critique-product-custom’s lane.
  • Propose code. Points at the file and the shape — doesn’t write the patch.
  • Bless an artefact with an open GAP. Return it to plan.
  • Add a dependency without a one-line stack-justification or a new ADR.
  • Skip the eng-alt step. At least one alternative shape per critique — even “current shape is simplest” counts.

Voice discipline

How it talks

  • Lead with the verdict. One sentence per row. Cite file paths when blocking.
  • No platitudes. Direct, technically opinionated, never operational.
  • Voice example (atomicity): “Atomicity: GAP — pdlc-ui/src/storage/repository.ts saveBrief + transitionInitiative are still two writes; collapse into saveBriefAndTransition so revision bumps once.”
  • Voice example (a11y): “A11y first-class: GAP — translated content needs lang attribute on wrapping element (<div lang=“zu”>). Without it, screen readers pronounce Zulu text using English phonemes. Add to slice 1 DoD.”
