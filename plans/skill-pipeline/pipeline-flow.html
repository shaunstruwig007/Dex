The skill pipeline

From an idea, to a build-ready PRD — in hours, not weeks.

Five skills, two repos, one journey. Each stage produces a typed artefact the next stage consumes — no whiteboards lost, no decisions buried in chat.

Owner: Design Manager
Skill catalogue: .claude/skills/
Vault: GitHub
Codebase: Bitbucket

Always-on context — loaded by every skill

Who we’re building for. Used to score strategic fit and to spot ICP-misalignment in discovery.

Weekly Market Intel

Friday brief synthesising newsletters, competitor moves, sector signals. Keeps discovery anchored to what changed.

06-Resources/Market_intelligence/synthesis/weekly/

Vault

Existing PRDs, meeting notes, person pages, decision logs. Used to detect duplicate work and stakeholder context.

The journey — idea to ship-ready

01

Idea

The spark

A natural-language hypothesis. “What if we let frontline employees see posts in their preferred language?” No structure required. Captured in chat, in a backlog, or on a sticky note.

in → chat / backlog
out → one paragraph

02

Discovery

What do we actually know?

/initiative-discovery-custom

Reads ICP, weekly intel, vault. Surfaces problem statement, user segments, evidence, evidence-gaps, stakeholders, candidate slices.

out → discovery doc
06-Resources/Product_ideas/
<name>_discovery.md

03

PRD

The spec, in slices

/prd-author-custom

bond_v1 shape: goal, metrics, requirements, slices (slice 1 = walking skeleton), test shape per slice, demo readiness, plan-mode-seed, build handoff.

out → PRD
06-Resources/PRDs/
<Feature>.md

04

Critique

Two-lens stress test

/critique-product-custom
/critique-engineering-custom

7-row product checklist (outcome, demo risk, honesty…) and 9-row engineering checklist (atomicity, idempotence, a11y, cheaper path…). Must-fixes loop back to PRD.

out → critique log
plans/skill-pipeline/sessions/
<date>-critiques.md

05

spec_ready

Green light to design & build

All must-fixes folded. PRD frontmatter flips to lifecycle: spec_ready. Body Status line mirrors state. The PRD is now safe to design against and safe to slice into tickets.

out → PRD with
critique_status:
must_fixes_folded

06

Design

Wireframes from PRD pointers

Claude Design (wireframe mode)

Consumes Design pointers + Failure-mode UX + A11y requirements. Produces wireframes that align with the slice plan. PRD frontmatter tracks design_pass_status.

out → wireframes
(in Claude Design)

07

Build handoff

PRD → tickets, in the codebase

PRD copied to the Bitbucket repo. Engineer pastes the plan-mode-seed block into Cursor Plan mode. Slices become tickets, in order, with test shape and demo-readiness attached.

out → tickets
in Cursor Plan mode
(Bitbucket repo)

↺ Critique loop. Must-fixes from stage 04 return to stage 03. The PRD author re-drafts — idempotently — and the loop runs until both critiques sign off. Two passes is normal. Three is a smell.

Why this matters

Before

A week of meetings. Ambiguous specs. Design debt. Decisions buried in Slack threads. Engineering re-discovering scope mid-sprint.

→
After

A few hours. A PRD with named slices, named tests, named risks. Design works from pointers. Engineering pastes one block into Plan mode and ships.
